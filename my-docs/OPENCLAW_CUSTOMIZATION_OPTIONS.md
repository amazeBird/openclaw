# OpenClaw 个人定制：诉求与可选实现路径

本文记录个人在「基于 OpenClaw 的智能系统」（如企微项目管理机器人）建设中的**诉求**与**可选方案**，便于后续按优先级落地。实现顺序建议：**先 Skill / 配置 / UI**，再考虑 **网关薄改**。

**仓库策略（已定案）**：以本 fork 持续演进为主、不强制同步上游；优先小改网关。见 `.cursor/rules/fariy-fork-strategy.mdc` 与 `FARIY.md`「Fork 策略」。

---

## 自建应用放哪：直接做在本工程 vs 另起仓库

这是你**最初问题**里「架构选址」的答案；下面「最优落地路径」是在**选定仓库之后**怎么排期实现。

### 默认推荐：**做在本仓库（fariy）里**

与当前 **Fork 策略**一致：你**不追求**与官方主线频繁同步，且要**经常改网关/UI/插件**时，把「自建应用」落在 **同一个 repo** 通常 **最省事、版本不漂移**。

| 做法                        | 说明                                                                                                |
| --------------------------- | --------------------------------------------------------------------------------------------------- |
| **通道 / 企微 / 对接 PM**   | 优先 **`extensions/<你的插件 id>/`**（或社区插件 + 本仓少量配置），与 OpenClaw 插件模型一致。       |
| **领域逻辑、脚本、小服务**  | 可放 **`packages/<name>/`**（若需独立包名/复用），或工作区目录 + Skill 调用 HTTP。                  |
| **单独控制台 / 另一套前端** | 可 **`apps/<你的应用名>/`**（与 `apps/macos` 等并列），通过 **稳定网关协议** 连本机网关；仍属单仓。 |
| **改主题与控制台**          | 继续 **`ui/`**（你已在做）。                                                                        |

原则：**用目录画出边界**，避免「自建」逻辑无命名地散落在 `src/` 各处；必须动核心时再动 **`src/gateway`** 等（见 Fork 规则）。

### 何时值得 **另起仓库**

- **应用与 OpenClaw 技术栈完全无关**（例如纯 Java 企微回调服务），且只通过 **HTTP/WebSocket 调网关**，不需要改插件 SDK。
- **要单独开源 / 单独 CI / 给团队只 checkout 应用仓库**，而网关固定用 **已安装的 `openclaw` 包**或固定镜像。
- **合规**：不允许把业务代码放进当前 monorepo。

另起仓库的代价：要自己维护 **与网关协议的版本对齐**、两套 CI、本地 **link / 私有 npm / 子模块** 等。

### 小结

- **你现在的路径**：**直接搭在本工程里**（`extensions` + `ui` + 必要时 `packages` / `apps`）是 **最优默认**。
- **另起工程**：适合「业务服务完全独立、只把 OpenClaw 当黑盒网关」的形态；**不是**你当前「要改 fork 能力」场景的首选。

### 服务器部署：要几个目录？

开发阶段「本工程 + 可选外部 workspace」在服务器上对应的是 **三类路径**（**不必**等于三个 Git 仓库；也**不必**一定拆成两个 clone，按你是否单独维护业务内容而定）。

| 类型                    | 放什么                                                                         | 典型路径示例                     | 说明                                                                                                                                                                           |
| ----------------------- | ------------------------------------------------------------------------------ | -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **运行时**              | 本仓库 **`pnpm build` 后的产物**（或 Docker 镜像内的 `openclaw`、内置控制 UI） | `/opt/fariy`、或仅容器镜像       | **一份** OpenClaw/fariy 安装；升级 = 重新 build / 拉新镜像。                                                                                                                   |
| **状态与配置（state）** | `openclaw.json`、会话存储、日志、credentials 等                                | `/var/lib/openclaw`              | 用环境变量 **`OPENCLAW_STATE_DIR`**（及必要时 **`OPENCLAW_CONFIG_PATH`**）指向；**与源码目录分离**，便于备份与重装运行时。                                                     |
| **工作区（workspace）** | `AGENTS.md`、Skills、助手要读写的项目文件（即你口中的「应用工程」内容）        | `/srv/openclaw-workspace/pm-bot` | 在 `openclaw.json` 里设 **`agents.defaults.workspace`** 或 **`agents.list[].workspace`** 为**绝对路径**；可以是 **另一个目录里 git pull 的仓库**，也可以是本机单独同步的目录。 |

**是否一定要「两个目录」？**

- **至少**：**运行时** + **state**（workspace 若很小也可先放在默认结构下，由配置决定）。
- **常见生产做法**：**运行时**（只读镜像） + **state 卷** + **workspace 卷** —— 这样「本工程」部署的是 **OpenClaw 二进制/镜像**，「应用工程」是 **workspace 那个目录**（第二个目录很常见，但它是 **数据/业务内容**，不是第二份 OpenClaw 源码）。

**Docker**：镜像承载运行时；`OPENCLAW_STATE_DIR` 与 workspace 目录 **挂卷** 持久化。官方若有 Docker 文档，以 `docs/install/docker` 为准。

**与开发机对应**：开发用 `~/.openclaw-dev`；上线改用服务器上的 **`OPENCLAW_STATE_DIR`** + 配置里的 **workspace**，不要用混 dev 与 prod 状态目录。

## 最优落地路径：搭建应用时边做边解决问题

本节是在 **选定「应用放在本仓库」** 之后，如何 **排期与分层** 的**综合答案**：在 **本 fork 上持续搭建**（企微 PM 机器人等）时，用**最少网关改动**、**最大复用 OpenClaw 现有能力**，把问题逐个关掉。细项仍见下文 §1–§5；目录分层见 §7。

### 三条原则

1. **先用自带能力** — 配置、`/status`、`/context`、`/usage`、`/compact`、**自动 compaction**（`docs/concepts/compaction.md`）能解决大半「爆窗、看不见用量」类问题，不必先写代码。
2. **按成本栈递进** — **配置 → Skill → `ui/` → `extensions/*` → `src/`（非 gateway）→ `src/gateway/`**。每一层能搞定就不下沉；与 `.cursor/rules/fariy-fork-strategy.mdc` 一致。
3. **可观测优先于改逻辑** — 遇到「卡住、不知道完没完」：先 **网关日志、`openclaw doctor`、run/会话标识**，区分 **模型慢 / 工具挂 / 上下文顶满**，再决定是调配置、限流工具还是改 UI 展示阶段。

### 分阶段做什么（推荐顺序）

| 阶段                  | 目标               | 主要动作                                                                                                                                  | 改动面                          |
| --------------------- | ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------- |
| **P0 地基**           | 开发闭环稳定       | `pnpm gateway:dev`、控制面连上、onboard/配置一致；`doctor` 无未处理硬伤                                                                   | 配置 / 环境                     |
| **P1 上下文与可见性** | 少断片、看得见窗口 | 打开 **compaction**；养成 **`/usage`** 或默认 usage footer；**UI 常显上下文条**（扩展 `renderContextNotice`，见 §2-B）                    | 配置 + 主要是 `ui/`             |
| **P2 业务与记录**     | PM / 企微 + 沉淀   | **通道**：社区企微插件或自建 `extensions`；**记录**：Skill + tool 写工作区或调 PM API（§3-A）；**交互**：TASKS → 欢迎 **chips**（§4-A）   | Skill、`extensions/`、`ui/`     |
| **P3 自动化**         | 少人工切换模式     | **chat / reasoner**：先用 **规则 + 指令 + Skill 文案**（§1-A）；不够再 **外部分类器 + hook**（§1-B）；定时摘要用 **cron + agent**（§3-B） | 小服务 + 可选 `src/` 薄改       |
| **P4 硬核体验**       | 续命、强观测       | **新 session + 摘要注入**（§2-D）；**UI 阶段条 / 超时**（§5-B）；仍不够再 **网关内状态机**（§5-C）                                        | 多涉及 `src/gateway` 或会话 API |

### 常见问题 → 首选动作（速查）

| 现象                      | 首选                                |
| ------------------------- | ----------------------------------- |
| 不知道上下文用了多少      | P1：`/status`、`/usage` + UI 常显条 |
| 长对话突然变差 / 截断     | P1：compaction；仍不够再考虑 §2-D   |
| 想自动选「轻聊 / 重推理」 | P3：§1-A；强需求再 §1-B / C         |
| 要把结论、待办记下来      | P2：§3-A Skill + 文件或 API         |
| 开场不想重复打字选任务    | P2：§4-A chips                      |
| 中途卡死                  | 原则 3；再 §5                       |

### 与下文各节的关系

- **上文「自建应用放哪」**：本仓库 vs 独立工程。
- §1–§5：每个诉求的 **A/B/C 可选方案**（展开细节）。
- §6：为何不能 **100%** 只靠 UI + Skill；**官方网关 + 自研 UI** 何时成立。
- §7：**`ui/` vs `src/gateway` vs 同进程后端** 目录归属。
- §8：与上表 **P1→P4 顺序一致** 的短列表，便于复制到任务表。

## 1. 按用户提问在「chat / reasoner」间路由

**诉求**：根据用户问题，用分析器决定走轻量对话还是强推理路径。

| 方案                           | 改动面                | 说明                                                                                                           |
| ------------------------------ | --------------------- | -------------------------------------------------------------------------------------------------------------- |
| **A. 规则 + 指令（推荐先做）** | 配置 / 文档 / Skill   | 在系统提示或 Skill 中约定场景；用户或助手用 `/thinking`、`/model` 等（以当前文档为准）切换。无代码或极少代码。 |
| **B. 外部分类器 + 会话覆盖**   | 小服务 + 可选网关钩子 | 小模型或规则服务返回标签；若存在「进主模型前」的 hook，只设 `thinkingLevel` / `reasoningLevel` / model 覆盖。  |
| **C. 网关内嵌路由**            | `src/` 回复链路       | 在 `get-reply` / directive 解析前插入分类器调用；合并上游成本高，仅在有强需求时做。                            |

## 2. 上下文长度可见 + 快满时总结并无缝延续

**诉求**：类似 Cursor 的上下文占用感知；快超限时自动总结并尽量「无缝」继续（避免新开对话从零开始）。

| 方案                           | 改动面          | 说明                                                                                                                                                           |
| ------------------------------ | --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **A. 配置 + 指令（推荐先做）** | 配置 + 使用习惯 | 启用 **`agents.defaults.compaction`**（见 `docs/concepts/compaction.md`）；使用 `/status`、`/context`、`/usage`、`/compact`（见 `docs/concepts/context.md`）。 |
| **B. 控制 UI 常显上下文条**    | `ui/`           | 已有 `renderContextNotice`（约 85% 起显示）；可改为**始终**展示 `used/limit` 进度，数据来自现有 session 字段。                                                 |
| **C. 同会话 compact**          | 已有能力        | 自动 compaction 减轻爆窗，**不新开 session**。                                                                                                                 |
| **D. 新 session + 摘要注入**   | 网关或工具链    | 真正「新窗口」需新建会话并把摘要写入首条消息；需明确 API（sessions 创建/切换 + 历史注入），属**中等改动**。                                                    |

**诉求**：运行中沉淀决策、结论、待办等。

| 方案                      | 改动面                 | 说明                                                         |
| ------------------------- | ---------------------- | ------------------------------------------------------------ |

| **B. Cron + agent**       | 配置 + 定时任务        | 周期性对会话做增量摘要追加。                                 |
| **C. 网关 hook 全量落盘** | `src/`                 | 每轮消息结构化日志；能力强，耦合大。                         |

## 5. 中途卡死、不可观测、需自修 bug

**诉求**：知道是否在跑、是否完成、为何卡住；能自行在 fork 里修。

| 方案                       | 改动面               | 说明                                                                                           |
| -------------------------- | -------------------- | ---------------------------------------------------------------------------------------------- |
| **A. 运维与文档**          | 无或极少             | 网关详细日志、`openclaw doctor`、`/status`、session 日志路径；区分模型慢 / 工具挂 / 上下文满。 |
| **B. UI 展示阶段与超时**   | `ui/` + 可能网关事件 | 显式「排队 / 工具执行 / 流式结束」与可取消；依赖网关是否暴露足够细的事件。                     |
| **C. 网关内状态机 / 超时** | `src/`               | 体验最可控，合并成本最高。                                                                     |

## 6. UI 层 vs Skill vs 网关：能否「只改 UI + Skill，网关用官方」？

### 6.1 是否大多数只要 UI 或 Skill？

**不完全是。**

- **Skill（+ 配置 + 文档）** 最适合：**领域知识、PM 接口、写工作区文件、Cron 触发的总结、通过自然语言引导用户下指令**。
- **UI** 最适合：**品牌与布局、上下文条、欢迎建议 chips、额外面板**（在**不改变网关协议语义**的前提下）。
- **网关** 通常在以下情况**绕不开**：
  - 进主模型**之前**自动改 `thinking` / `reasoning` / `model`（无用户指令、无现成 hook）；
  - **自动新建 session 并注入摘要**的闭环；
  - **全量消息 hook**、与渠道强绑定的路由或安全策略；
  - 修复**官方 bug**或协议行为与你的 UI 假设不一致时。

结论：**大量体验可以 UI + Skill 解决**；**「自动路由模型 / 自动换 session 续命」类**往往至少要**薄改网关**或等上游能力。

### 6.2 能否 UI 自定义、网关用官方 npm？

**可以，这是一种有效架构**，前提是：

1. **自定义 UI 只使用稳定网关协议**（WebSocket / HTTP RPC、会话、聊天、工具流式事件等），不把 `fariy` 里未合并上游的私有字段当成依赖。
2. **行为差异**尽量放在：你自己的 **Skill、外部服务、UI 内展示逻辑**，而不是改协议。
3. 若你需要 **fork 才有的 UI 数据**（例如额外 session 元数据），而官方网关不提供，则要么 **放弃该特性**，要么 **仍跑 fork 网关**或 **自己加一层 BFF**。

**拆分后的典型形态**：官方/自建网关进程 + 独立部署的 **自定义控制台前端**（或企微侧只作为 channel，由官方网关收消息）。

**当前 `fariy` 仓库**：是 **单体**（UI + 网关同源）；若改为「官方网关 + 自研 UI」，需要维护 **协议兼容性** 与 **构建/部署分离**。

## 7. 当前仓库：控制 UI、网关与相关服务端目录归属

> 口语里的「网关」有时指整个 Node 后端；下表把 **狭义网关**（`src/gateway`）与 **同进程服务端** 分开写，避免和 **`ui/` 控制台** 混淆。

### 7.1 属地控制 UI 层（浏览器里的控制台）

| 路径                         | 作用                                                                                      |
| ---------------------------- | ----------------------------------------------------------------------------------------- |
| **`ui/`**                    | 主 Web 控制台：Lit + Vite。入口如 `ui/src/main.ts`、`ui/src/ui/app.ts`、`app-render.ts`。 |
| **`ui/src/ui/views/`**       | 各 Tab 页面：聊天、概览、通道、Agent、配置、用量、Cron、会话等。                          |
| **`ui/src/ui/chat/`**        | 聊天区渲染、分组消息、导出、slash 等 **纯前端** 逻辑。                                    |
| **`ui/src/ui/controllers/`** | 浏览器里调网关的 **客户端状态与请求**（不是服务端）。                                     |
| **`ui/src/i18n/`**           | 控制台文案与语言。                                                                        |
| **`ui/src/styles*.css`**     | 控制台样式（含 urban 等主题）。                                                           |
| **`ui/public/`**             | 控制台静态资源（logo、头像等）。                                                          |

这些代码 **不跑模型、不接 Telegram/企微等通道**；只通过 **WebSocket / HTTP** 连到本机网关进程。

### 7.2 属地网关层（狭义：`src/gateway`）

| 路径               | 作用                                                                                              |
| ------------------ | ------------------------------------------------------------------------------------------------- |
| **`src/gateway/`** | WebSocket 服务、HTTP 辅助接口、`chat.*` / `sessions.*` 等 **协议实现**、控制 UI 的托管与 CSP 等。 |

控制台里的发送、拉历史、abort 等，最终落在上述 RPC / 方法上。

### 7.3 服务端但常与网关同进程（口语里常统称「后端 / 网关能力」）

跑在 **Node** 里、与 `openclaw gateway` **同一进程树** 加载，**不属于** `ui/`：

| 路径                                                        | 作用                                                          |
| ----------------------------------------------------------- | ------------------------------------------------------------- |
| **`src/agents/`**                                           | Agent 执行、工具调用、与模型交互等 **核心智能逻辑**。         |
| **`src/auto-reply/`**                                       | 各通道进来后的自动回复、指令、thinking/reasoning 等解析链路。 |
| **`src/sessions/`**                                         | 会话存储与生命周期（与网关 `sessions` RPC 配合）。            |
| **`src/channels/`**、**`src/routing/`**、**`src/pairing/`** | 通道抽象、路由、配对。                                        |
| **`src/config/`**                                           | `openclaw.json` 等配置与 schema。                             |
| **`src/commands/`** + **`src/cli/`**                        | CLI 子命令（`doctor`、`channels`、`agent` 等）。              |
| **`extensions/*`**                                          | 插件（含部分 channel），由运行时加载。                        |
| **`src/plugins/`**、**`src/plugin-sdk/`**                   | 插件加载与对外 SDK。                                          |

若说「网关层 = 自建系统依赖的后端」，通常 **至少包含 `src/gateway` 与上表**；**只有 `ui/` 算纯 UI 层**。

### 7.4 其它客户端（也算 UI，但不是 `ui/` 这个 Web 包）

| 路径                                                    | 说明                                                                   |
| ------------------------------------------------------- | ---------------------------------------------------------------------- |
| **`apps/macos/`**、**`apps/ios/`**、**`apps/android/`** | 菜单栏 / 移动端 **原生壳**：界面在本仓库，业务仍多通过网关或共享逻辑。 |

### 7.5 一句话边界

- **只改展示、主题、按钮、进度条、欢迎 chips** → 主要在 **`ui/`**。
- **改「连上控制台之后发生的事」**（会话、模型、工具、通道、配置落盘）→ **`src/gateway` 起，并牵涉 `src/agents`、`src/auto-reply`、`extensions` 等**，**不是** UI 层。
